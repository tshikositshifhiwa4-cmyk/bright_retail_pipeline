1. Company Background

BrightLearn is a South African retail chain operating five stores across the country. The business sells products across five categories: Groceries, Electronics, Clothing, Home & Kitchen, and Beauty & Personal Care. BrightLearn operates a customer loyalty programme with three tiers — Bronze, Silver, and Gold — which rewards repeat purchases.

BrightLearn's five stores are located in Sandton (Gauteng), Cape Town (Western Cape), Umhlanga (KwaZulu-Natal), Pretoria (Gauteng), and Gqeberha (Eastern Cape). Each store runs its own point-of-sale (POS) system, which records all customer transactions.

2. Business Problem

The Business Intelligence team has been unable to produce any meaningful reports for the senior leadership team because BrightMart's transactional data has never been properly organised or stored in a queryable format.

Currently, the POS system exports all sales data as a single flat file. This file contains all transaction details, customer details, product details, and store details mixed together in one table — with no normalisation, no consistent formatting, and known data quality issues that have accumulated over time.

The Chief Executive Officer (CEO), Rofhiwa, has escalated this as a business-critical issue. Without a functional data warehouse, BrightLearn cannot make data-driven decisions about inventory, customer retention, or store performance.

CRITICAL: This is a real business problem with real consequences. The BI team is blocked. Leadership is waiting. This assignment simulates the exact environment you will encounter as a professional Data Engineer. Your deliverables must be production-quality.

3. Source Data

You have been provided with one CSV file:

File name: BrightLearn_Raw_Data.csv
Location: data/BrightLearn_Raw_Data.csv
Description: A single flat, denormalised export from the BrightMart POS system, covering sales transactions from January 2026 to June 2026. All entity data (customers, products, stores, transactions) is mixed into one file. One row represents one product line item within a transaction.

This file is the only data source you are permitted to use. You may not supplement it with any other data. You must work with what you have been given, including its imperfections.

WARNING: The source data contains known and unknown data quality issues. It is your responsibility as the Data Engineer to identify, document, and handle all anomalies before loading data into the warehouse. The business cannot trust reports built on dirty data.

5. Scope of Work — What You Must Deliver

You are required to design and build a complete data engineering solution. The deliverables are defined below. How you achieve them is your decision. You must make all technical, architectural, and data quality decisions independently.

5.1 GitHub Repository

  • Create a public GitHub repository named: brightlearn-de-exam-[yourfirstname]-[yourlastname]
  • All code, scripts, documentation, and screenshots must be committed to this repository
  • Your commit history will be reviewed — you must have a minimum of one commit per working day
  • Commit messages must be meaningful and descriptive. Generic messages ("update", "final", "done") will result in mark deductions
  • The repository must be accessible to your facilitator by the submission deadline
  • Your README.md must explain what your project does and the order in which scripts should be run

5.2 SQL Server Database (SSMS)

  • Design and create a normalised relational database in SQL Server to store the BrightLearn data
  • The database must support all 8 business questions listed in Section 4
  • You must define appropriate primary keys, foreign keys, and data types for every table
  • Your database design decisions must be documented — include a data model diagram or description in your repository
  • All T-SQL scripts used to create the database schema must be committed to GitHub
  • All T-SQL analytical queries answering BQ-01 through BQ-08 must be committed to GitHub

5.3 SSIS ETL Package

  • Build an SSIS package that loads data from BrightLearn_Raw_Data.csv into your SQL Server database
  • The SSIS package must perform all necessary transformations — including data cleansing, type casting, deduplication, and business rule application — before loading data into the destination tables
  • The SSIS package file (.dtsx) must be committed to your GitHub repository
  • The package must be executable on a clean database without manual intervention
  • Any SSIS configurations (connection strings, file paths) must be parameterised so that the package can be run from any machine

5.4 Data Quality Report

  • Submit a written Data Quality Findings Report as a file in your GitHub repository
  • The report must list every data quality issue you discovered in the source file
  • For each issue, document: what the problem is, how many records are affected, and what decision you made to resolve it
  • This report demonstrates your ability to think critically about data before building on it

5.5 Analytical Queries

  • Write T-SQL queries that answer all 8 business questions (BQ-01 to BQ-08)
  • Queries must be saved as individual .sql files in your repository
  • Each query file must include a comment at the top referencing the business question it answers
  • Queries must return correct, meaningful results when executed against your populated database

7. Out of Scope

The following items are explicitly not required and will not be assessed:

  • Building any dashboard, visualisation, or front-end reporting interface
  • Real-time or streaming data pipelines
  • Cloud deployments (Azure, AWS, GCP)
  • Python, R, or any programming language other than T-SQL for data transformation
  • Data for periods outside January 2026 – June 2026
  • Inventory restocking workflow automation
